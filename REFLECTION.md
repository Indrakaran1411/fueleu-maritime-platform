# Reflection on AI-Assisted Development

Using AI agents on this project made one distinction very clear: agents are strong at structure, repetition, and scaffolding, but they are not reliable sources of truth for domain-specific rules. In this FuelEU compliance system, the architecture, repository patterns, test scaffolding, and UI boilerplate came together much faster with agent help. At the same time, the actual compliance formulas and regulatory thresholds still required manual verification against the source regulation. The most important lesson was that AI output is useful only when treated as a draft that must be reviewed, tested, and corrected by the engineer.

The main efficiency gain compared with manual coding was speed on repetitive work. Generating interfaces, handlers, SQL queries, DTO mapping, React tab structure, and test skeletons was significantly faster than writing them from scratch. Work that might normally take several hours could often be reduced to a focused review-and-correct cycle. The gains were smaller in business logic, where a wrong constant or edge-case assumption can invalidate the result. In those areas, manual coding was sometimes just as fast once the time spent validating and fixing agent output was included.

If I were doing this again, I would improve the workflow in three ways. First, I would use spec-first prompting by giving the agent the relevant regulation excerpt before asking for domain logic. Second, I would write or tighten tests earlier so the agent is coding against explicit behavior instead of vague intent. Third, I would keep the workflow more incremental by documenting prompts and committing after each major agent-assisted step. That would make it easier to trace which outputs came from the agent, catch mistakes earlier, and reduce rework caused by context drift.
